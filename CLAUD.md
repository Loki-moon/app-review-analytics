# App Review Intelligence Lab
현행화 구현 가이드

## 1. 문서 목적

이 문서는 신규 MVP를 처음 설계하는 문서가 아니라, 현재 구현되어 있는 Streamlit 기반 App Review Intelligence Lab 프로젝트를 기준으로
UI와 플랫폼 조회 범위 그리고 상태 관리 구조를 현행화하기 위한 구현 가이드이다.

Claude Code는 새로운 제품을 처음부터 다시 설계하지 말고,
현재 코드베이스를 최대한 유지하면서 아래 요구사항만 우선 반영한다.

핵심 원칙은 다음과 같다.

- 이미 구현된 구조를 최대한 재사용한다
- 기존 분석 탭과 파이프라인 구조를 유지한다
- 상단 고정 영역 UI를 레퍼런스에 맞게 정돈한다
- 플랫폼 조회 범위는 Google Play Store와 App Store까지 확장한다
- 불필요한 다크모드와 과도한 재구성은 하지 않는다
- 크레딧을 많이 쓰는 대규모 리팩터링 대신, 필요한 부분만 명확히 수정한다

---

## 2. 현재 코드 기준 현황 요약

현재 프로젝트에는 아래 요소가 이미 구현되어 있다.

- Streamlit 메인 앱 진입 구조
- input 페이지와 result 페이지 분리
- 상단 앱 선택 pill 형태 헤더
- 리뷰 기간 및 플랫폼 선택 컨트롤 바 렌더 구조
- 결과 탭 구조
- 리뷰 수집 이후 파이프라인 실행 구조
- 검증 탭 포함 결과 렌더 구조
- 라이트 톤 중심 CSS

따라서 이번 작업은 아래 방향으로 진행한다.

- 없는 기능을 새로 과도하게 추가하지 않는다
- 이미 있는 구조를 기준으로 UI와 스크래퍼 라우팅만 정리한다
- '새 제품 PRD'처럼 장황하게 재설계하지 않는다

---

## 3. 이번 현행화 범위

이번 현행화의 범위는 아래 4가지이다.

### 3-1. 상단 고정 UI 정리
첨부 캡처와 레퍼런스를 참고하여 상단 고정 영역을 정돈한다.

### 3-2. 플랫폼 조회 범위 확장
Google Play Store와 App Store는 실제 조회 가능 상태로 맞춘다.

### 3-3. 라이트모드 전용 정리
다크모드 관련 상태와 UI는 제거하거나 비활성화한다.

### 3-4. 문서와 코드의 기준 통일
현재 코드와 문서가 어긋나는 부분을 맞춘다.

---

## 4. 상단 고정 영역 요구사항

상단 고정 영역은 첨부한 캡처 이미지와 아래 레퍼런스를 참고하여 구현한다.

참고 레퍼런스
- https://some.co.kr/analysis/compare/association

구성은 아래와 같다.

### 4-1. 헤더
- 분석할 앱 입력 영역
- 최대 5개 앱까지 선택 가능
- 앱이 2개 이상이면 가운데 `VS` 표시
- 비교 앱 추가용 `+` 버튼 제공
- 첫 입력창 placeholder는 `분석할 앱을 추가하세요`
- 두 번째 이후 입력창 placeholder는 `비교할 앱을 추가하세요`

### 4-2. 상단 필터 바
- 시작일
- 종료일
- 빠른 기간 선택 버튼
    - 1개월
    - 3개월
    - 6개월
    - 12개월
    - 24개월
    - 36개월
- 앱 마켓 플랫폼 토글
    - Google Play Store
    - App Store
    - One Store
    - Galaxy Store

### 4-3. 플랫폼 기본 상태
- Google Play Store 기본 On
- App Store 기본 Off
- One Store 기본 Off
- Galaxy Store 기본 Off

### 4-4. 고정 방식
- 헤더와 필터 바는 스크롤 시에도 상단에 유지되도록 구현한다
- 현재 화면에서 가장 중요한 제어 영역은 상단 고정 영역이다
- 본문 결과 영역보다 상단 조작 경험을 우선 정리한다

## 5. 플랫폼별 처리 기준

### 5-1. 이번 현행화에서 실제 조회 지원
- Google Play Store
- App Store

위 2개 플랫폼은 실제 앱 조회 및 리뷰 조회가 가능해야 한다.

### 5-2. UI 노출만 유지
- One Store
- Galaxy Store

위 2개는 토글 UI는 유지하되, 실제 조회 미구현이면 안내 문구를 노출한다.

예시 문구
- 현재 버전에서는 해당 플랫폼 조회가 제한될 수 있어요

### 5-3. 구현 원칙
- 스크래퍼 구조는 플랫폼 라우팅 방식으로 정리한다
- 분석 시작 시 선택된 플랫폼을 순회하면서 각 플랫폼 스크래퍼를 호출한다
- 현재처럼 Google Play로 하드코딩하지 않는다
- 앱 검색 단계에서도 플랫폼별 검색 결과를 구분해 보여줄 수 있어야 한다

## 6. 디자인 및 CSS 기준

디자인은 some.co.kr 비교분석 상단 UI의 밀도와 정렬감을 참고한다.
다만 완전 복제가 아니라, Streamlit 안에서 유사한 톤으로 정돈하는 수준으로 구현한다.

### 6-1. 모드 정책
- 다크모드는 지원하지 않는다
- 라이트모드만 지원한다

### 6-2. 컬러 기준
- 헤더 배경: linear-gradient(135deg, #6C5CE7 0%, #8B7CF8 100%)
- 페이지 배경: #F4F5FB
- 카드 배경: #FFFFFF
- 기본 텍스트: #1A1A2E
- 보조 텍스트: #6B7280
- 테두리: #E5E7EB
- 활성 포인트 컬러: #6C5CE7

### 6-3. 스타일 방향
- 라운드 입력창
- 얇은 테두리
- 과하지 않은 그림자
- 밝은 배경 위 정보 밀도 높은 배치
- 검색창과 토글이 한 화면에서 즉시 읽히는 구조

### 6-4. CSS 원칙
- 기존 CSS 파일을 최대한 유지하면서 상단 고정 영역 위주로 수정한다
- 전역 색상은 라이트 기준으로 통일한다
- dark_mode 분기용 CSS는 제거하거나 무시한다

## 7. 상태 관리 현행화

세션 상태는 현재 구조를 최대한 유지하되 아래 기준으로 정리한다.

### 유지
- selected_apps
- selected_platforms
- start_date
- end_date
- raw_df
- pipeline_result
- validation_result
- analysis_done

### 정리 대상

### 기본값

- end_date = 오늘
- selected_apps = []

## 8. 분석 실행 로직 수정 기준

현재 분석 실행은 Google Play Store 스크래퍼 고정 호출이 아니라
선택 플랫폼 기준 분기 처리로 변경한다.

### 수정 방향
- 선택된 앱별로
- 선택된 플랫폼별로
- 대응하는 scraper를 가져와 조회한다

### 처리 원칙
- Google Play Store와 App Store는 실제 조회 수행
- One Store와 Galaxy Store는 미구현 시 skip + warning
- 수집된 리뷰는 platform 컬럼 기준으로 통합 저장
- 결과 화면은 기존 파이프라인 구조를 그대로 사용한다

### 중요
- 기존 `run_pipeline`, `run_all_validations`, 결과 탭 렌더 구조는 최대한 유지한다
- 수집부만 플랫폼 확장 구조로 정리한다
- 분석 결과 렌더 로직을 전면 재작성하지 않는다

## 9. 앱 검색 및 선택 UX 기준

### 앱 선택 규칙
- 최대 5개 앱 선택 가능
- 동일 앱 중복 선택 불가
- 플랫폼별 검색 결과를 구분해서 보여준다
- 사용자가 앱을 확정 선택한 뒤에만 selected_apps에 반영한다

### 검색 결과 카드
- 앱 아이콘
- 앱 정식 이름
- 마켓명
- app_id
- 개발사명
- 평점 또는 설치 수
- `이 앱이 맞아요` 버튼

### 예외 처리
- 검색 결과 없음
- 플랫폼 미지원
- 동일 앱 중복 선택
- 최대 5개 초과 시도

## 12. Claude Code 작업 우선순위

### 1순위
상단 고정 헤더와 필터 바를 첨부 캡처 기준으로 정돈한다.

### 2순위
라이트모드 전용으로 정리하고 dark_mode 관련 상태와 UI를 제거한다.

### 3순위
Google Play Store와 App Store를 실제 조회 가능하도록 스크래퍼 라우팅을 수정한다.

### 4순위
앱 검색 결과와 플랫폼 선택 상태가 자연스럽게 연결되도록 입력 UX를 다듬는다.

### 5순위
기존 결과 페이지와 분석 파이프라인 연결이 깨지지 않는지 점검한다.

## 13. 구현 시 유의사항

- 새로운 구조를 처음부터 만들지 말고 현재 코드베이스를 기준으로 수정할 것
- UI는 상단 고정 영역 완성도를 우선할 것
- App Store는 실제 조회 가능 대상으로 포함할 것
- Google Play Store와 App Store를 동일 레벨의 지원 플랫폼으로 취급할 것
- 나머지 플랫폼은 추후 확장 여지를 남기되 이번 작업 범위에 과도하게 포함하지 말 것
- 문서 표현에서도 'MVP를 새로 만든다'는 표현보다 '현행화'와 '정리' 중심으로 작성할 것

## 14. 바로 수행할 작업

1.
현재 상단 앱 선택 영역을 첨부 캡처처럼 고정 헤더 구조로 정리한다.

2.
헤더 아래 필터 바에 시작일, 종료일, 빠른 기간 버튼, 플랫폼 토글을 배치한다.

3.
다크모드 관련 세션 상태와 UI를 제거하고 라이트모드만 유지한다.

4.
분석 실행 시 선택된 플랫폼 기준으로 scraper를 분기 호출하도록 수정한다.

5.
Google Play Store와 App Store는 실제 앱 조회 및 리뷰 조회가 가능하도록 연결한다.

6.
기존 결과 탭 구조와 분석 파이프라인은 유지한 채, 입력 경험과 플랫폼 수집 범위만 우선 현행화한다.

7.
작업 완료 후 수정한 파일 목록과 변경 포인트를 함께 정리한다.
